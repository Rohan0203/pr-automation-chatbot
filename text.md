## PR Automation Chatbot вҖ” Backend

### Purpose
A conversational AI chatbot that helps Cargill engineers generate **AWS infrastructure YAML configs** (S3, Glue DB, IAM) through chat, then automatically creates **GitHub Pull Requests** in the `object-provisioning` repo.

### Architecture
```
Frontend (React) вҶ’ FastAPI вҶ’ LLM (Groq/OpenAI/Azure/TrueFoundry) вҶ’ GitHub API (PR creation)
                           вҶ’ SQLite DB (sessions, messages, state)
```

### Entry Points
| File | Role |
|------|------|
| main.py | FastAPI app factory вҖ” lifespan, CORS, route mounting |
| app/api/routes.py | Chat endpoints (`POST /api/chat`, history, PR creation) |
| app/auth/routes.py | GitHub OAuth flow |

### Agent System (State Machine)
```
IDLE вҶ’ STRUCTURED_FLOW (Q1вҶ’Q2вҶ’Q3) вҶ’ COLLECTING вҶ’ AWAITING_CONFIRMATION вҶ’ REVIEWING вҶ’ BATCH_PROMPT вҶ’ PR_SETUP вҶ’ DONE
```

| Module | Responsibility |
|--------|---------------|
| orchestrator.py | Thin dispatcher вҖ” routes to correct handler based on session phase |
| structured_flow_handler.py | Guided wizard: environment вҶ’ resource type вҶ’ enterprise вҶ’ conditionals |
| collector_agent.py | LLM-driven field extraction/validation using resource MD guides |
| confirmation_handler.py | YAML preview вҖ” confirm/edit/cancel |
| reviewer_agent.py | Policy validation against org rules |
| batch_handler.py | Accumulates multiple resources per PR |
| pr_handler.py | Branch creation, commit, cross-fork PR |
| field_deriver.py | Deterministic derived fields (bucket_name, account_id, etc.) |
| prompts.py | All prompt templates |

### Data Flow (Happy Path)
1. User message вҶ’ intent classification
2. Structured flow (button-driven steps)
3. Text collection of remaining fields via LLM
4. Deterministic field derivation
5. LLM generates YAML
6. Syntax + structural validation
7. User confirmation
8. Organizational policy review
9. Batch accumulation
10. Fork вҶ’ branch вҶ’ commit вҶ’ cross-fork PR

### Key Services
| Service | Purpose |
|---------|---------|
| llm_client.py | Unified LLM client (Groq/OpenAI/TrueFoundry/Azure) |
| schema_registry.py | Loads MD field guides as LLM context |
| scm_adapter.py | Fork-based GitHub workflow via PyGithub |
| path_resolver.py | Resolves commit paths based on account + resource type |

### Validation вҖ” Two Layers
- **Field-level** (collector): each field valid per the MD guide
- **Policy-level** (reviewer): complete YAML complies with organizational rules in `data/*-validations/`

### Database (SQLite + async SQLAlchemy)
Tables: `chat_sessions`, `chat_messages`, `resource_states`, `pr_records`

### Auth
GitHub Enterprise OAuth (`git.cglcloud.com`) вҖ” token stored per session, supports web app and Chrome extension flows.

### Design Principles
- **Markdown files = single source of truth** for field definitions and rules
- **LLM does all extraction/validation/generation** вҖ” Python is a thin state machine
- **No hardcoded validation** вҖ” LLM interprets MD guides at runtime

## My Proposal for Simplified State + Collection

Instead of 6 phases, we do:

```
IDLE вҶ’ SELECTING вҶ’ COLLECTING вҶ’ CONFIRMING
```

- **SELECTING**: Ask constrained-option questions one by one (env вҶ’ resource вҶ’ enterprise вҶ’ subgroup вҶ’ conditionals). Each question has explicit options. State tracks which questions remain.
- **COLLECTING**: LLM-driven free-text extraction for remaining fields. Context = only the fields we still need + their validation rules (minimal tokens).
- **CONFIRMING**: Show YAML, confirm/edit/cancel.

For **context to the LLM** during COLLECTING, we don't send the whole 300-line MD file. We build a **focused context** listing only:
- Fields still needed
- Validation rule per field (one line each)
- Fields already known (so LLM doesn't re-ask)

## Agent Responsibilities (Clean Separation)

| Agent | Input | Output | LLM? |
|-------|-------|--------|------|
| **Parser** | Raw user message | Detected intent + extracted values | Yes (or keyword match for simple cases) |
| **Context Loader** | Resource types | Field specs (name, required, options, validation) | No вҖ” data lookup |
| **Collector** | Field specs + current state | List of missing fields per resource | No вҖ” pure diff logic |
| **Formatter** | Missing fields + their metadata | Natural conversational message | Yes вҖ” makes it human-friendly |
| **Extractor** | User reply + context of what was asked + field specs | Extracted key-value pairs | Yes вҖ” understanding natural language |

**Key insight:** Only 3 of 5 steps need LLM. Collector and Context Loader are pure logic вҖ” fast, deterministic, testable.

## What Each Agent Does

### 1. Intent Agent (`agents/intent.py`) вҖ” LLM
```
IN:  user message + glossary context
OUT: {
       resources: ["s3", "glue_db"],
       extracted: {"enterprise_or_func_name": "AGTR", "data_env": "dev"}
     }

Then: create Resource objects, load field_specs, pre-fill extracted values
      вҶ’ phase = COLLECTING
```

### 2. Collector (`agents/collector.py`) вҖ” Pure Logic
```
IN:  resource.field_specs + resource.fields (current state)
OUT: list of fields still needed (filtering out: already filled, derivable,
     dependency not met)

Example:
  field_specs has 15 fields for glue_db
  3 already filled, 4 derivable, 1 depends on unknown field
  вҶ’ returns 7 askable fields
```

### 3. Formatter (`agents/formatter.py`) вҖ” LLM
```
IN:  missing fields (from collector) + their descriptions/options +
     what's already known (for context)
OUT: natural language message asking the user for values

Example:
  "I still need a few details:
   вҖў Intake ID (format: M followed by digits)
   вҖў Source system name (lowercase, underscores, e.g. sap_finance)
   вҖў Data owner email
   вҖў Data owner GitHub username
   вҖў Data leader ID"
```

### 4. Extractor (`agents/extractor.py`) вҖ” LLM
```
IN:  user message + previous bot question + field specs for what was asked
OUT: [
       {"field": "intake_id", "value": "M0000934", "valid": true},
       {"field": "source_name", "value": "sap finance", "valid": false,
        "issue": "no spaces allowed, use underscores"},
     ]

Then: valid values вҶ’ resource.fields[name] = value
      invalid вҶ’ carry forward, formatter includes correction next round
```

### 5. Deriver (`agents/deriver.py`) вҖ” Pure Logic
```
IN:  resource.fields (all collected values)
OUT: derived values: {
       "database_name": "lh_sap_raw_dev",
       "database_s3_location": "s3://dev-lh1-agtr-src/raw/...",
       "aws_account_id": "578647603827",
       "aws_region": "us-east-1"
     }

Runs after collector says "all required filled"
```

## Agent Architecture: How To Structure It

There are 3 common patterns:

### Option A: Many Independent Agents (LangChain style)
```
Each agent has its own system prompt, memory, LLM call
Intent Agent вҶҗвҶ’ LLM
Extractor Agent вҶҗвҶ’ LLM
Formatter Agent вҶҗвҶ’ LLM
Generator Agent вҶҗвҶ’ LLM
```
**Problem:** Too many LLM calls per turn. User sends one message вҶ’ 3-4 LLM calls in sequence. Slow, expensive.

### Option B: One Big Agent With Tools (OpenAI function-calling style)
```
One agent with one system prompt
Tools: extract_fields(), check_missing(), derive_fields(), generate_yaml()
Agent decides which tool to call
```
**Problem:** Agent has to be smart enough to pick the right tool at the right time. Over-reliant on LLM reasoning. Unpredictable.

### Option C: Code Orchestrator + Targeted LLM Calls (What We Should Do) вң“
```
Orchestrator (PURE CODE вҖ” not LLM) decides what to do
Makes ONE targeted LLM call per user turn with the right prompt
Pure-code functions handle everything else
```

### The 3 LLM Functions (Not Agents вҖ” Just Prompts)

| Function | When Called | Prompt Focus | Output |
|----------|-----------|--------------|--------|
| **`llm.parse(message, field_specs)`** | Every user message during COLLECTING | "Given these expected fields and this user message, extract values + validate" | `{fields: {name: value}, invalid: {name: issue}, intent: "answer" \| "question" \| "edit" \| "drop"}` |
